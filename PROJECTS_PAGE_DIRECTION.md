# Projects Page Direction

## Goal

Create a dedicated projects page that turns the current mix of portfolio work, internal utilities, product demos, and side projects into a clearer body of work for recruiters and hiring managers.

The page should show the strongest work quickly, then let interested visitors inspect more detail without forcing every project into a long case study.

## Audience

- Recruiters who need a fast read on role fit.
- Hiring managers who want proof of product thinking, execution, and technical translation.
- Teammates or collaborators who want to understand the kind of tools I build.

## Positioning

Primary message:

> Tools that turn workflow friction into usable product systems.

Supporting idea:

I identify messy operational workflows, translate them into clearer product requirements, design the useful path, and build working tools that make the workflow easier to repeat.

## Recommended Structure

1. Hero / Orientation
   - Short positioning headline.
   - One paragraph explaining the kind of work represented.
   - Small proof chips for focus areas: product systems, ad tech operations, learning UX, workflow tooling.

2. Featured Case Studies
   - Three large, always-visible case studies.
   - Recommended first set:
     - Kumu: personal learning product and mobile-first sentence practice.
     - Spotlight: Washington Post ad product discovery and recommendation microsite.
     - Workshop: project planning and product-shaping workspace.
   - Each should include:
     - Strong headline.
     - Concise context.
     - Problem / Built / Signal proof points.
     - Visual or live-feeling preview.
     - Primary link.

3. Project Index
   - Compact filterable grid.
   - Each card should provide the tool logo, a one-line value prop, type/status metadata, tags, project link, and a case-study action.
   - Case-study depth should open in one shared full-width panel below the grid rather than expanding inside individual cards. This preserves the grid rhythm and avoids uneven card-height jumps.

4. Build Method
   - The method should live near the top of the page as small supporting context, not as a large repeated section at the bottom.
   - The three-part pattern is enough:
     - Find friction.
     - Shape the system.
     - Build the path.

## Homepage Implication

Do not update the homepage yet.

If this page works, the homepage can later be simplified into a shorter "Selected Work" section with a CTA to `/projects/`. The homepage should make the strongest impression; the projects page should carry the depth.
